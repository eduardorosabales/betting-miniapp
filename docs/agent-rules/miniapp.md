# `betting-miniapp` — Stack, arquitectura e invariantes

> Extraído **verbatim** de `instrucciones.txt` (v4.5, ahora `.bak`). Referencias `§N` aluden a ese respaldo.

## Stack técnico (§2.2)
| Capa | Tecnología | Notas |
|---|---|---|
| Marcado | HTML5 | Sin framework SPA |
| Estilos | CSS embebido en `<style>` | Sin Tailwind ni preprocessors |
| Lógica | JavaScript vanilla (ES2020+) | Sin React/Vue/Svelte |
| Gráficos | Chart.js (CDN) | Confirmar versión exacta antes de proponer cambios |
| SDK Telegram | `telegram-web-app.js` (CDN oficial Telegram) | Provee `window.Telegram.WebApp` |
| Build | **Ninguno** | Lo que se commitea es lo que se sirve |
| Storage cliente | `sessionStorage` por defecto | `localStorage` prohibido salvo la excepción acotada del caché IA — INV-MINI-04 |

**Deuda silenciosa:** introducir un build step (Webpack/Vite/esbuild), un framework SPA, o `localStorage` requiere análisis explícito — el costo de operación crece y rompe el modelo "edita HTML, push, deploy".

## Arquitectura del repo (§3.2)
```
betting-miniapp/
├── index.html            ← (~3260 líneas) Chart.js + Telegram WebApp SDK + lógica UI
├── .gitignore            ← cubre .env, *.session (repo PÚBLICO — INV-MINI-10/11)
├── README.md             ⚠ Placeholder de 1 línea — completar hosting/URL (TODO-6)
└── assets/
    ├── favicon-32.png
    └── apple-touch-icon-180.png
```

**Reglas (no negociables):**

1. **`index.html` jamás conoce la estructura de la BD.** Consume contratos JSON estables de `/api/*`.
2. **No hay build step.** Lo commiteado es lo servido. Cualquier herramienta de transformación rompe esa propiedad.
3. **Toda lógica de cálculo de negocio vive en el bot, no en la miniapp.** La miniapp es presentación + interacción, no cálculo. Excepción documentada: la regex de normalización `Total O/U` que sí está duplicada (ver `INV-XCUT-02`).

## Invariantes de la Mini-App (§4.6, `INV-MINI`)
- **INV-MINI-01:** Sin build step. El `index.html` que se commitea es el que se sirve. Cualquier transformación intermedia (Webpack/Vite/Babel) debe justificarse como cambio estructural.
- **INV-MINI-02:** Toda lógica de cálculo de negocio vive en `betting-stats-bot` y se consume vía `/api/*`. La miniapp solo presenta y formatea. Excepción documentada: la regex `Total O/U` (ver `INV-XCUT-02`).
- **INV-MINI-03:** Auth desde la miniapp: **únicamente** header `X-Telegram-Init-Data` con `window.Telegram.WebApp.initData`. El header legacy `X-Api-Key` fue retirado (TODO-2): se eliminaron `getSecret`/`saveSecret`/`handleSecretPaste`, el input de clave y el banner ahora dice "Abre esta app desde Telegram". `apiHeaders()` solo envía `X-Telegram-Init-Data`.
- **INV-MINI-04:** Almacenamiento cliente en `sessionStorage` **por defecto**. `localStorage` está **prohibido** (vector XSS persistente) salvo **una única excepción documentada**: el caché del análisis de Patrones IA, clave `betstats_patrones_cache` (`_PATRONES_LS_KEY`, helpers `_cargarPatronesGuardados`/`_guardarPatrones`). Justificación: Telegram borra `sessionStorage` al cerrar el webview, lo que forzaría un análisis Opus nuevo en cada reapertura (coste real + consumo del presupuesto Anthropic, INV-AI-04). Condiciones que mantienen segura la excepción y que **deben** preservarse: (1) solo se persiste el texto del análisis + su timestamp — **nunca** tokens, credenciales ni `initData`; (2) el contenido se renderiza con `esc()` + `renderPatronesMarkdown()` (parser que escapa antes de formatear), nunca por `innerHTML` crudo. Cualquier OTRO uso de `localStorage` sigue prohibido y requiere ampliar esta excepción aquí primero. Se refuerza en `INV-SEC-03`.
- **INV-MINI-05:** Charts vía Chart.js cargado por CDN. Cambiar de librería (a Recharts, ApexCharts, etc.) es decisión arquitectónica que requiere análisis del impacto en performance y bundle.
- **INV-MINI-06:** El contrato JSON de `/api/*` se considera estable. Cualquier cambio en el bot (renombrar campos, modificar tipos, agregar/eliminar endpoints) requiere actualización coordinada en la miniapp en PRs simultáneas. Romper este contrato sin tocar la miniapp = pantalla en blanco silenciosa.
- **INV-MINI-07:** Toda imagen subida a /api/analyze-ticket se pre-comprime en cliente con compressImage() antes del envío. La función aplica re-compresión adaptativa sobre una cadena de presets (1920px q=0.85 → 1600px q=0.75 → 1280px q=0.70) hasta que el blob baje del umbral _MAX_BLOB_BYTES_TO_SEND = 6 MB. Si ningún preset alcanza ese umbral, la función rechaza localmente con error claro; nunca se envía un body que el bot fuera a rechazar por client_max_size. El umbral cliente (6 MB) debe mantenerse estrictamente menor que client_max_size del bot menos el overhead JSON (~10 MB − 20 chars). Cualquier endpoint nuevo que reciba imágenes debe usar la misma función helper.
- **INV-MINI-08:** Toda condición de error en cargarDatos() (y handlers de fetch equivalentes) debe terminar en un estado UI visible al usuario, sin paths que dejen el loader inicial estático en pantalla. AbortError no es excepción — se renderiza como timeout con botón de reintento. Aplicado el [fecha del deploy].
- **INV-MINI-09:** La <meta http-equiv="Content-Security-Policy"> del index.html debe enumerar exactamente los orígenes externos que el propio HTML consume. Antes de mergear cualquier PR que añada un <script src="…">, <link href="…">, <img src="…"> externo o un fetch a un nuevo dominio, la CSP correspondiente (script-src, style-src, font-src, connect-src, img-src) debe actualizarse en la misma PR. La directiva frame-ancestors no se declara en <meta> (los user agents la ignoran ahí); debe entregarse como header HTTP o no declararse en absoluto.
- **INV-MINI-10:** Prohibición absoluta de credenciales hardcodeadas.** Dado que el repositorio de la Mini-App es público en GitHub, queda estrictamente prohibido introducir tokens de bots de Telegram, llaves de API de Anthropic, credenciales de bases de datos o secretos de Railway (`MINIAPP_SECRET`) en cualquiera de los archivos del repositorio. Cualquier configuración de entorno requerida debe ser provista de forma dinámica por el backend, inyectada mediante meta-tags configurables en el HTML, o solicitada dinámicamente en runtime al usuario.
- **INV-MINI-11:** Restricciones de integridad en archivos y dependencias locales. Se debe mantener el repositorio libre de archivos de entorno locales (como `.env`), sesiones de desarrollo o credenciales de test locales. No se deben introducir frameworks SPA pesados o pasos de construcción que compliquen el flujo directo estático ("edita HTML, push, deploy"), manteniendo el proyecto libre de vectores de exposición en GitHub.
- **INV-MINI-12:** Las media queries de CSS (especialmente la de diseño responsivo de escritorio `@media(min-width:768px)`) deben colocarse siempre estrictamente al final del bloque `<style>` en `index.html`. Esto evita que las declaraciones base (móvil) que se definen después en el flujo de cascada terminen sobreescribiendo y rompiendo los estilos de escritorio.
- **INV-MINI-13:** Prohibición absoluta de eventos inline en el HTML. Todo comportamiento interactivo estático o dinámico se asocia exclusivamente mediante delegación de eventos en los listeners globales de `click`, `input`, `change` y `paste` (`document.addEventListener(...)`), interactuando con atributos `data-action` y otros atributos `data-*` semánticos en los elementos DOM. Esto mitiga permanentemente los riesgos de inyección XSS en atributos interpolados y centraliza la lógica interactiva.
- **INV-MINI-14:** Sección **"Motor de Neto Mensual"** en el tab Patrones (`renderNetoMensual`), render de **tarjetas/chips** que consume el campo `neto_mensual` de `/api/patterns` (ver INV-XCUT-09). Presentación pura (INV-MINI-02): todo input pasa por `esc()`, sin eventos inline (INV-MINI-13), barra de concentración en CSS puro (**sin Chart.js**, decisión consciente — añadir un chart aquí requeriría análisis por INV-MINI-05). **Debe tolerar la ausencia del campo** `neto_mensual` (clientes o análisis previos sin él): si falta, está vacío o trae `error`, `renderNetoMensual` retorna `""` y el texto IA se renderiza igual. El número grande de cada tarjeta de mes es el **pico neto intra-mes (`neto_max`)**, etiquetado "pico" (no el neto final), y el resumen "Mejor mes" usa también el pico; ambos con fallback a `neto` final si un análisis cacheado no trae `neto_max`. El color de "mejor mes" se condiciona al signo del valor mostrado (no asumir verde). El CSS `.nm-*` reusa la paleta existente y se ubica en el bloque base, nunca tras las media queries (INV-MINI-12).
- **INV-MINI-15:** **Navegación mobile-first de 4 grupos + sub-nav + BackButton nativo** (rediseño 2026-06-05). La barra inferior expone **4 grupos** alcanzables con el pulgar (`NAV_GROUPS`: Resumen · Gráficas · Historial · Gestión), nunca los 12 tabs sueltos con scroll horizontal del diseño anterior. Cada grupo agrupa una o más de las **12 secciones existentes** (sus render fns en `_RENDER_FNS` **se conservan intactas** — solo cambia el acceso, cero métricas eliminadas), mostradas como *segmented control* (`renderSubNav`, chips `.sub-chip`); la sub-nav se oculta si el grupo tiene una sola sección. Los análisis avanzados (`ADVANCED`: Kelly · Capital · Patrones IA) viven en una sub-vista "Más análisis" que activa el **`BackButton` nativo** (`tg.BackButton`, `onClick → closeAdvanced`), con chip "‹ Volver" de respaldo fuera de Telegram. `showTab(id)` deduce el grupo desde la sección y sincroniza por sí solo nav inferior, sub-nav y BackButton (firma sin parámetro `btn`; ningún call-site pasa botón). Cambios de grupo/sección emiten `HapticFeedback` (helper `haptic()`). Todo control interactivo táctil mide **≥44×44px** (refresh, `.mes-nav`, `.filter-chip`, `.range-chip`, `.sub-chip`, retry, "Cargar gráfica"). Toda regla CSS nueva (`.sub-nav/.sub-chip/.hero-*/.range-*/.bet-badge*/.skeleton/.stale-banner`) vive en el bloque base **antes** de las media queries (INV-MINI-12).
- **INV-MINI-16:** **Tema heredado del cliente Telegram (claro/oscuro).** `applyTelegramTheme()` mapea `tg.themeParams` + `tg.colorScheme` a las variables CSS de superficie/texto (`--bg/--surface/--card/--card2/--text/--text-2/--border/--text-3` y el "chrome" `--header-bg/--nav-bg/--nav-bg-solid`), de modo que el header y la barra inferior siguen el modo del cliente (no fondo oscuro fijo). Los **colores semánticos** (`--win` verde / `--loss` rojo / `--pend` ámbar / `--accent`) **NO** derivan del tema: significan lo mismo en claro y oscuro. Chart.js es theme-aware: ticks/grids/leyendas/tooltips leen `cssVar('--chart-tick'|'--chart-grid'|'--text-2'|'--card2')`; nunca colores hex hardcodeados en las configs. `tg.onEvent('themeChanged')` re-aplica el tema y reconstruye las gráficas visibles (`refreshChartsTheme`). Sin `themeParams` (app fuera de Telegram o cliente viejo) se conserva la paleta oscura por defecto del `:root`. Header/fondo nativos vía keywords `setBackgroundColor('bg_color')`/`setHeaderColor('secondary_bg_color')` (auto-siguen el tema), no hex fijos.
- **INV-MINI-17:** **Caché de `/api/stats` en `sessionStorage`** (clave `betstats_stats_cache`, helpers `_loadStatsCache`/`_saveStatsCache`/`_applyStats`). Permite **pintado instantáneo** al reabrir/refrescar y siempre refresca en segundo plano; **evita re-render** (parpadeo/reset de scroll) si el JSON fresco es idéntico al cacheado. Si el fetch falla **con** caché en pantalla, se muestra un banner "datos guardados" (`_showStaleBanner`) sin romper la vista; en carga **en frío** sin caché se muestran **skeleton loaders** (`_renderSkeleton`, clase `.skeleton`) — nunca pantalla en blanco (refuerza INV-MINI-08). Cumple **INV-MINI-04**: usa `sessionStorage` (permitido), **no** `localStorage`. El payload cacheado son stats agregadas del propio usuario (sin secretos); render sigue pasando por `esc()`.
- **INV-MINI-18:** **`MainButton` nativo como acción primaria del modal de Gestión** (`_setupMainButton`/`_teardownMainButton`). Al abrir el modal de alta/edición se muestra el `tg.MainButton` con el texto de la acción y su `onClick → submitBet()`; el botón propio `#mSubmitBtn` se **oculta** cuando hay MainButton y permanece como **fallback** fuera de Telegram (o clientes sin la API). Durante el guardado se usa `showProgress()/disable()` y al terminar/errar `hideProgress()/enable()`, con `HapticFeedback` `success`/`error`. `closeModal()` hace **teardown** (`offClick` + `hide`) para no apilar handlers entre aperturas. No interfiere con el `BackButton` (que solo se usa en la sub-vista de análisis, INV-MINI-15).

- **INV-MINI-19:** **Subresource Integrity (SRI) en el script de Chart.js cargado por CDN.** El `<script src="…cdnjs…/Chart.js/4.4.1/chart.umd.min.js">` lleva `integrity="sha512-CQBWl4fJHWbryGE+Pc7UAxWMUMNMWzWxF4SQo9CgkJIN1kx6djDQZjh3Y8SZ1d+6I+1zze6Z7kHXO7q3UyZAWw=="` + `crossorigin="anonymous"` + `referrerpolicy="no-referrer"`. Motivo (auditoría de ciberseguridad): el repo es estático/público y depende de un CDN de terceros (cadena de suministro, CIBERSEGURIDAD §5); el hash garantiza que el navegador **rechace** el script si cdnjs lo sirviera alterado. El hash se verificó contra el SRI oficial de cdnjs **y** contra el archivo realmente servido (sha512 idénticos). **Atado a la versión:** si se sube/baja la versión de Chart.js (hoy 4.4.1, INV-MINI-05), **hay que recalcular el `integrity`** o la carga romperá; calcularlo con `curl -s <url> | openssl dgst -sha512 -binary | openssl base64 -A`. El SDK `telegram-web-app.js` de telegram.org **no** lleva SRI a propósito (Telegram lo actualiza sin versionar; un hash fijo lo rompería) — la CSP `script-src` restringida a `telegram.org` (INV-MINI-09) es su control. No quitar el `integrity` de Chart.js ni dejarlo desactualizado tras un cambio de versión.

## Variables / configuración (§6.3)
Como es estática (HTML/JS sin build), **no tiene variables de entorno propias**. La configuración viaja por dos vías:

1. **Hardcoded en el HTML:** la URL del backend del bot (ej. `https://betting-stats-bot.up.railway.app`). Cualquier cambio de URL requiere PR + redeploy de la miniapp.
2. **Provisto por Telegram en runtime:** `window.Telegram.WebApp.initData`, `initDataUnsafe.user`, etc.

**Si en el futuro la miniapp necesita configuración por entorno**, las opciones aceptables son: (a) un archivo `config.json` separado servido por GitHub Pages, (b) parámetros en la URL del WebApp configurada en BotFather. Introducir un build step solo por esto está prohibido por `INV-MINI-01`.

Si se introduce una variable nueva en cualquier servicio, **debe documentarse aquí en la misma propuesta**.

## Resiliencia y manejo de errores (§7.2)
- **Errores HTTP de la API:** mostrar mensaje user-friendly vía `safeJson()` (mensajes por status, sin exponer parseos crudos del proxy). Reintento automático **solo en 503/504** vía `fetchRetry503()` en las lecturas (`/api/stats`, `/api/patterns`); las mutaciones no se reintentan (no idempotentes).
- **`window.Telegram.WebApp` no disponible:** la app degrada a la pantalla de gestión con banner "Abre esta app desde Telegram" (estado UI visible, no fallo silencioso). Tras retirar `X-Api-Key` (TODO-2), el CRUD requiere `initData` — la app solo es funcional abierta desde Telegram.
- **`fetch` con timeout:** envuelto en `AbortController`, **12s** por default; `AbortError` se renderiza como timeout con botón de reintento (INV-MINI-08).

- **INV-MINI-20:** **CSP endurecida: `script-src` SIN `'unsafe-inline'` (JS externalizado) y `connect-src` al host EXACTO del backend (audit M-3/B-3).** (1) Los 2 bloques `<script>` inline pasaron a archivos estáticos `assets/boot.js` (no-blur pre-paint, en `<head>`) y `assets/app.js` (la app, fin de `<body>`), de modo que `script-src` ya **no** lleva `'unsafe-inline'`: la CSP ahora **bloquea cualquier `<script>` inline inyectado** (mitigación real de XSS por inyección, defensa en profundidad sobre el escapado de `esc()`). Externalizar JS **no** rompe INV-MINI-01: **no es build step**, sigue siendo estático edita-y-publica; los `.js` se sirven same-origin (`'self'`). **A partir de ahora el JS de la app se edita en `assets/app.js`, no dentro del HTML.** `style-src 'unsafe-inline'` **se mantiene** (estilos `style=""` inline masivos; riesgo de seguridad mucho menor que script). (2) `connect-src` pasó del comodín `https://*.up.railway.app` al **host exacto** `https://betting-stats-bot-production.up.railway.app` (el del meta `api-url`), reduciendo el destino de exfiltración ante una hipotética inyección (coherente con INV-MINI-09: la CSP enumera orígenes exactos). **Verificado:** `node --check` de ambos `.js` + smoke-test headless (Chrome) — la app renderiza igual que con el JS inline (el `TypeError reading 'resumen'` fuera de Telegram/sin backend es **preexistente**, idéntico en el HTML original). No reintroducir `'unsafe-inline'` en `script-src` ni el comodín en `connect-src`; recalcular nada de SRI (los `.js` propios van same-origin, sin integrity).
